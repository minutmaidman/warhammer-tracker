# Warhammer Roster Tracker

A single-file hobby tracker for Warhammer 40,000 and Age of Sigmar armies. Tracks units, points, painting progress, loadouts, and wargear.

---

## How `index.html` is structured

The file has four layers, top to bottom.

---

### Layer 1 — External dependencies (lines 9–11)

```html
<script src="https://unpkg.com/react@18/...">
<script src="https://unpkg.com/react-dom@18/...">
<script src="https://unpkg.com/@babel/standalone/...">
```

Three CDN scripts loaded from the internet. React is the UI framework. Babel lets you write JSX directly in the browser without a build step. **The app requires an internet connection on first load** to fetch these — after that the browser caches them.

---

### Layer 2 — CSS (lines 12–393)

One `<style>` block. Organised roughly as:

- **CSS variables** (`--bg`, `--gold`, `--red` etc.) — change colours here to retheme the whole app
- **Layout** — sidebar, main content area, KPI bar
- **Components** — buttons, inputs, badges, modals, the unit table
- **Loadout panel** — everything `.loadout-*`, `.enh-*`, `.weapon-*`, `.kw-*`, `.ability-*`

To add a new visual component, add its CSS here.

---

### Layer 3 — Baked data (lines 397–402)

#### `window.UNITS_DATA` (line 398, one long line)

The unit database — every faction, unit, tier, and point cost for both 40K and AoS.

```js
{
  gameSystems: { wh40k: {...}, aos: {...} },
  factions:    { wh40k_SM: { id, name, gameSystem }, ... },
  units:       [ { id, name, factionId, role, tiers: [{models, points}], wahapediaUrl, msrpCAD }, ... ]
}
```

To add or update units, edit this object. It is sourced from Wahapedia — refreshing it would need its own build script similar to `build-loadout.mjs`.

#### `window.LOADOUT_DATA` (lines 400–402, between the markers)

Generated by `node build-loadout.mjs`. **Never edit manually** — regenerate from CSVs.

```js
{
  enhancements:       { "000001234": { name, cost, detachment, desc, faction } },
  unitEnhancements:   { 1472: ["000001234", ...] },   // unit id → enhancement ids
  unitAbilities:      { 10: [{ name, type, phase, desc }] }, // AoS unit id → abilities
  unitWargear:        { 882: [{ name, type, range, A, BSWS, S, AP, D, desc }] },
  unitWeapons:        { 10: [{ name, type, rng, atk, hit, wnd, rnd, dmg, abilities }] },
  unitDefaultLoadout: { 882: "Every model is equipped with: guardian spear." }
}
```

---

### Layer 4 — React application (lines 404–1289)

One `<script type="text/babel">` block. Babel compiles the JSX at page load. Everything is plain JavaScript functions.

#### Helpers (lines 406–438)

Small utility functions used everywhere:

| Function | Purpose |
|---|---|
| `uid()` | Generates a random ID for new list/unit entries |
| `clamp(v, min, max)` | Keeps a number within a range |
| `paintColor(pct)` | Returns a CSS colour based on paint percentage |
| `loadState()` / `saveState()` | Read/write all app state to `localStorage` under key `wh_roster_v1` |

#### Sub-components (lines 441–495)

Small reusable UI pieces:

| Component | Purpose |
|---|---|
| `Toast` | Notification popup (bottom-right corner) |
| `PaintPct` | Coloured percentage display in the unit table |
| `UnitSearchInput` | Autocomplete dropdown in the Add Unit modal |

#### Modals (lines 497–710)

Self-contained popup windows. Each takes props and calls a callback when done:

| Component | Purpose |
|---|---|
| `AddUnitModal` | Search or manual unit entry — builds the unit object that gets saved |
| `NewListModal` | Create a new list (name, game system, faction, points cap) |
| `MsrpModal` | Edit the CAD retail price on a unit |

#### Keyword tooltip (lines 712–753)

| Symbol | Purpose |
|---|---|
| `KW40K` | Dictionary of 40K weapon keyword definitions |
| `kwLookup(raw)` | Normalises a keyword string (strips numeric suffixes, handles `anti-X Y+`) to find it in the glossary |
| `KwTooltip` | Renders a keyword with a hover popup if a definition exists — purple dotted underline indicates a defined keyword |

To add a new keyword definition, add an entry to `KW40K`.

#### LoadoutPanel (lines 755–904)

The expandable panel under each unit row. Receives `entry`, `gameSystem`, `listId`, `onUpdate`. Renders four sections in order:

1. **Enhancements** — 40K CHARACTER units only. Radio buttons grouped by detachment. Selecting one adds its point cost to the unit total.
2. **Unit Abilities** — AoS units only. Click-to-expand ability cards with type badges and rules text.
3. **Wargear checkboxes** — Both systems. Check a weapon to mark it as built; expands to show the full stat line with keyword tooltips.
4. **Loadout Notes** — Always shown. Free-text field pre-filled with the unit's default equipment on add.

#### App (lines 905–1288)

The root component — owns all state and renders the full page layout.

**State:**
- `state` — all lists and units, persisted to `localStorage`
- `expandedUnits` — which unit rows currently have their panel open (not persisted)
- `showAddUnit`, `showNewList`, `msrpEntry`, `toast`, `search` — UI flags

**List operations:** `addList`, `deleteList`, `updateListName`

**Unit operations:** `addUnit`, `updateUnit`, `removeUnit` — passed down to child components as props

**KPI calculation** (`useMemo`) — derives total points (including enhancement costs), model count, MSRP, and paint percentage from the active list. Recalculates whenever the list changes.

**Render structure:** sidebar (list selector) → main area (KPI bar → toolbar → search → unit table → modals)

#### Mount (line 1289)

```js
ReactDOM.createRoot(document.getElementById('root')).render(<App/>);
```

Starts the app by mounting `App` into `<div id="root">`.

---

## Data flow

```
localStorage → loadState() → App state → props to child components
                                ↑
user actions → addUnit / updateUnit / removeUnit → saveState() → localStorage
```

React re-renders the UI automatically whenever state changes.

## Common maintenance tasks

| Task | Where |
|---|---|
| Change a colour | CSS variables, lines 13–34 |
| Add a new table column | `<thead>` ~line 1034, and the `filteredUnits.map` row ~line 1060 |
| Add a field to unit entries | `AddUnitModal.handleAdd` ~line 500, plus anywhere that reads it |
| Add a keyword definition | `KW40K` object, line 713 |
| Update loadout / wargear data | Run `node build-loadout.mjs` from the project folder |
| Update faction / unit data | Edit `window.UNITS_DATA` on line 398 |

## Rebuilding loadout data

When CSV files are updated, run:

```
node build-loadout.mjs
```

This reads the following files from the project folder:

| File | Used for |
|---|---|
| `Enhancements.csv` | 40K character enhancement names and point costs |
| `Datasheets_enhancements.csv` | Which units can take which enhancements |
| `Datasheets_wargear.csv` | 40K weapon profiles per unit |
| `Datasheets.csv` | 40K default loadout text per unit |
| `Warscrolls_abilities.csv` | AoS unit abilities |
| `Warscrolls_weapons.csv` | AoS weapon profiles per unit |

The script updates `index.html` in place between the `<!-- LOADOUT-DATA-BEGIN -->` and `<!-- LOADOUT-DATA-END -->` markers. Only that block changes — everything else in the file is untouched.

---

## Deployment

The app is a single file. Push `index.html` to GitHub Pages and it works. The CSV files and `build-loadout.mjs` are excluded by `.gitignore` — they stay local as build inputs only.
