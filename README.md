# Warhammer Roster Tracker

A single-file hobby tracker for Warhammer 40,000 and Age of Sigmar armies. Tracks units, points, painting progress, wargear loadouts, leader attachments, and unit stats. Includes a Game Day view optimised for tablets and printing.

---

## How `index.html` is structured

The file has four layers, top to bottom.

---

### Layer 1 — External dependencies

```html
<script src="https://unpkg.com/react@18/...">
<script src="https://unpkg.com/react-dom@18/...">
<script src="https://unpkg.com/@babel/standalone/...">
```

Three CDN scripts loaded from the internet. React is the UI framework. Babel lets you write JSX directly in the browser without a build step. **The app requires an internet connection on first load** — after that the browser caches them.

---

### Layer 2 — CSS

One `<style>` block. Organised roughly as:

- **CSS variables** (`--bg`, `--gold`, `--red` etc.) — change colours here to retheme the whole app
- **Layout** — sidebar, main content area, KPI bar
- **Components** — buttons, inputs, badges, modals, unit table
- **Loadout panel** — `.loadout-*`, `.enh-*`, `.weapon-*`, `.kw-*`, `.ability-*`, `.stat-*`
- **Game Day view** — `.gd-*` cards and responsive grid
- **Print styles** — `@media print` block that switches to light mode and hides hobby columns

---

### Layer 3 — Baked data

#### `window.UNITS_DATA` (one long line)

The unit database — every faction, unit, tier, and point cost for both 40K and AoS.

```js
{
  gameSystems: { wh40k: {...}, aos: {...} },
  factions:    { wh40k_SM: { id, name, gameSystem }, ... },
  units:       [ { id, name, factionId, role, tiers: [{models, points}], wahapediaUrl, msrpCAD }, ... ]
}
```

To add or update units, edit this object. Sourced from Wahapedia — refreshing it would need its own build script similar to `build-loadout.mjs`.

#### `window.LOADOUT_DATA` (between the `<!-- LOADOUT-DATA-BEGIN / END -->` markers)

Generated by `node build-loadout.mjs`. **Never edit manually.**

```js
{
  enhancements:       { "000001234": { name, cost, detachment, desc, faction } },
  unitEnhancements:   { 1472: ["000001234", ...] },       // 40K unit id → enhancement ids
  unitAbilities:      { 10: [{ name, type, phase, desc }] }, // AoS unit id → abilities
  unitWargear:        { 882: [{ name, type, range, A, BSWS, S, AP, D, desc }] }, // 40K weapons
  unitWeapons:        { 10: [{ name, type, rng, atk, hit, wnd, rnd, dmg, abilities }] }, // AoS weapons
  unitLeaderTargets:  { 2472: [894, 903, 909, ...] },     // 40K leader id → valid bodyguard unit ids
  unitStats40k:       { 882: [{ name, M, T, Sv, invSv, W, Ld, OC }] }, // per-model stat profiles
  unitStatsAos:       { 10: { M, Save, Ctrl, HP, Ward } },
  unitDefaultLoadout: { 882: "Every model is equipped with: guardian spear." }
}
```

### Layer 4 — React application

One `<script type="text/babel">` block. Babel compiles JSX at page load. Everything is plain JavaScript functions.

#### Helpers

| Function | Purpose |
|---|---|
| `uid()` | Generates a random ID for new list/unit entries |
| `clamp(v, min, max)` | Keeps a number within a range |
| `paintColor(pct)` | Returns a CSS colour based on paint percentage |
| `loadState()` / `saveState()` | Read/write all app state to `localStorage` under key `wh_roster_v1` |

#### Sub-components

| Component | Purpose |
|---|---|
| `Toast` | Notification popup (bottom-right corner) |
| `PaintPct` | Coloured percentage display in the unit table |
| `UnitSearchInput` | Autocomplete dropdown in the Add Unit modal |

#### Modals

| Component | Purpose |
|---|---|
| `AddUnitModal` | Search or manual unit entry — builds the unit object saved to state |
| `NewListModal` | Create a new list (name, game system, faction, points cap) |
| `MsrpModal` | Edit the CAD retail price on a unit |

#### Keyword tooltip

| Symbol | Purpose |
|---|---|
| `KW40K` | Dictionary of 40K weapon keyword definitions |
| `kwLookup(raw)` | Normalises a keyword string (strips numeric suffixes, handles `anti-X Y+`) |
| `KwTooltip` | Renders a keyword with a hover popup if a definition exists — purple dotted underline |

To add a new keyword definition, add an entry to the `KW40K` object.

#### GameDayCard

Renders a single unit as a self-contained card for the Game Day view. Receives `entry`, `gameSystem`, `listUnits`. Shows, in order:

- Role badge + points (including enhancement cost)
- Unit name + qty if > 1
- Leader attachment indicators (↳ attached to / ⚔ led by)
- Stat block (same labelled cells as the panel)
- Selected enhancement — name, cost, and full description text
- Checked weapons — each as a named block with a full stat line below it

Used only in Game Day mode — no interaction, purely presentational.

#### LoadoutPanel

The expandable panel under each unit row in the standard table view. Receives `entry`, `gameSystem`, `listId`, `listUnits`, `onUpdate`. Renders sections in order:

1. **Stat block** — Always shown if data exists. Compact labelled cells: `M / T / SV / W / LD / OC` (40K) or `Move / Save / Ctrl / HP / Ward` (AoS). Multi-profile units show a row per model type.
2. **Attached to** — 40K units with leader data only. Dropdown filtered to valid bodyguard units already in the current list. Selecting one saves `attachedTo` on the entry and shows indicators in both rows.
3. **Enhancements** — 40K CHARACTER units only. Detachment dropdown + radio buttons. Selecting one adds the point cost to the unit's total and the KPI bar.
4. **Unit Abilities** — AoS units only. Click-to-expand ability cards with type badges (colour-coded) and full rules text.
5. **Wargear** — Both systems. Each weapon is a checkbox row; checking one expands a stat line with keyword tooltips for 40K.

#### App

The root component — owns all state and renders the full page layout.

**State:**

| State | Purpose |
|---|---|
| `state` | All lists and units — persisted to `localStorage` |
| `expandedUnits` | Set of unit entry IDs with open panels (not persisted) |
| `gameDay` | Boolean toggle between table view and Game Day card view |
| `sortBy` | Current sort key — `'default'`, `'name'`, `'role'`, `'pts-desc'`, `'pts-asc'`, `'paint'` |
| `showAddUnit`, `showNewList`, `msrpEntry`, `toast`, `search` | UI flags |

**Derived data (useMemo):**

| Value | Purpose |
|---|---|
| `kpis` | Total points (including enhancement costs), models, MSRP, paint % |
| `leaderMap` | `{ bodyguardEntryId → leaderEntry }` — used to render "⚔ Leader name" indicators |
| `filteredUnits` | Active list units filtered by search and sorted by `sortBy`. In default order, leaders are grouped immediately before their attached bodyguard unit. Both the table view and Game Day view consume this value. |

**List operations:** `addList`, `duplicateList`, `deleteList`, `updateListName`

- `duplicateList(id)` — deep-copies the list and all its units (each gets a new `uid`), appends `(Copy)` to the name, and makes the copy the active list.

**Unit operations:** `addUnit`, `updateUnit`, `removeUnit` — all update `state` and trigger `saveState()`.

**Render structure:** sidebar (list selector with ⧉ duplicate and ✕ delete per list) → KPI bar → toolbar (with Game Day toggle) → search + sort → table or card grid → modals.

#### Mount

```js
ReactDOM.createRoot(document.getElementById('root')).render(<App/>);
```

## Game Day view

Click **⚔ Game Day** in the toolbar to switch from the full hobby table to a clean card grid.

- **3 columns** on laptop → **2 columns** on tablet → **1 column** on phone
- Each card shows stat block, leader link, full enhancement description, and checked weapons with stat lines
- Hobby columns (painted, built, MSRP etc.) are hidden
- Respects the active sort and search filter — the same order and subset visible in the table appears in the cards
- **Print-ready**: `Ctrl+P` from Game Day mode renders in light mode with 3 columns and no sidebar

## Sorting

The sort dropdown sits in the search row. Options:

| Option | Behaviour |
|---|---|
| **Order: Added** (default) | Original add order, with leaders grouped immediately before their attached bodyguard |
| **Order: Role** | Alphabetical by role, then by name within each role |
| **Order: Name** | A → Z |
| **Order: Pts ↓** | Most expensive first (includes enhancement cost) |
| **Order: Pts ↑** | Cheapest first |
| **Order: Least Painted** | Lowest paint % first — useful for hobby queue planning |

## Data flow

```
localStorage → loadState() → App state → props to child components
                                ↑
user actions → updateUnit / addUnit / etc. → saveState() → localStorage
```

React re-renders automatically whenever state changes.

## Unit entry schema

Each unit in a list is a plain object with these fields:

```js
{
  id:                 string,   // random uid, roster-entry identifier
  unitId:             number|null, // id from UNITS_DATA (null for manual entries)
  name:               string,
  role:               string,   // "Characters", "Battleline", "Infantry", etc.
  wahapediaUrl:       string,
  points:             number,   // base points (before enhancement)
  baseModelCount:     number,
  tierIdx:            number,
  tiers:              [{models, points}],
  qty:                number,   // how many times this unit appears in the list
  msrpCAD:            number|null,
  acquired:           number,
  built:              number,
  primed:             number,
  painted:            number,
  selectedEnhancement: string|null,  // enhancement id key from LOADOUT_DATA.enhancements
  attachedTo:         string|null,   // id of the bodyguard roster entry this leader is attached to
  loadoutChecked:     string[],      // weapon names checked in the wargear panel
}
```

## Common maintenance tasks

| Task | Where |
|---|---|
| Change a colour | CSS variables at the top of the `<style>` block |
| Add a table column | `<thead>` and the `filteredUnits.map` row in `App` |
| Add a field to unit entries | `AddUnitModal.handleAdd` (both search and manual paths) |
| Add a keyword definition | `KW40K` object in the React script block |
| Add a Game Day card section | `GameDayCard` component |
| Update loadout / wargear / stats data | Run `node build-loadout.mjs` |
| Update faction / unit data | Edit `window.UNITS_DATA` |

## Rebuilding loadout data

When CSV files are updated, run from the project folder:

```
node build-loadout.mjs
```

This reads all of the following files:

| File | Used for |
|---|---|
| `Enhancements.csv` | 40K character enhancement names and point costs |
| `Datasheets_enhancements.csv` | Which 40K units can take which enhancements |
| `Datasheets.csv` | 40K unit names, roles, and default loadout text |
| `Datasheets_wargear.csv` | 40K weapon profiles per unit |
| `Datasheets_models.csv` | 40K per-model stat profiles (M/T/SV/W/LD/OC) |
| `Datasheets_leader.csv` | 40K leader → valid bodyguard unit compatibility |
| `Warscrolls_abilities.csv` | AoS unit special abilities |
| `Warscrolls_weapons.csv` | AoS weapon profiles per unit |
| `Warscrolls.csv` | AoS unit stat profiles (Move/Save/Ctrl/HP/Ward) |

The script updates `index.html` in place between the `<!-- LOADOUT-DATA-BEGIN -->` and `<!-- LOADOUT-DATA-END -->` markers. Only that block changes — everything else in the file is untouched.

## Deployment

The app is a single file. Push `index.html` to GitHub Pages and it works. The CSV files and `build-loadout.mjs` are excluded by `.gitignore` — they stay local as build inputs only.
